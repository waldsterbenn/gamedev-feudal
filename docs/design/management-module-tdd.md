# Technical Design Document (TDD) — Management Module

---

## 1. Terminology

Terms specific to the Management Module. Global project terminology (GameModule, Context Passing, Bootstrap Sequence, Coordinator Pattern, etc.) is defined in `module-workflow-and-docs.md`, `game-coordinator-module-tdd.md`, and `architecture-overview.md` and is not repeated here.

* **Populant**: An NPC worker that is visible in the 3D world and participates in the management simulation. Within the management context, physical characters are referred to exclusively by this term.
* **ZoneNode**: The fundamental data unit of the management world map — a discrete territory point (a clearing, node, or economic hub) that can be controlled, settled, and worked. Completely inert while at `WILDERNESS` tier.
* **Settlement Tier**: The administrative state of a `ZoneNode` (e.g., `WILDERNESS`, `CAMP`, `VILLAGE`, `TOWN`). Determines which buildings and jobs are available.
* **Stockpile**: The localized commodity inventory belonging to a single `ZoneNode`. Production deposits here; consumption and construction deduct from here.
* **Qualification Profile**: The skill label string registered by a `ManagementPopulantComponent` (e.g., `"unskilled"`, `"woodcutter"`). Used during workforce allocation matching.
* **Manual Job Postings**: Environmental jobs manually opened by the player on a node via the UI (e.g., `"forager"`, `"woodcutter"`), as distinct from jobs generated by completed buildings.
* **Labor Pool / Workforce Matching**: The tick-time process of assigning available Populants to open job slots according to the `JobPriorities` hierarchy.
* **Canopy Density**: The current physical volume of standing wood on a node (`0.0`-`1.0`). Depleted by woodcutters; governs mushroom foraging yields and ecosystem interactions.

## 2. Domain Objects

### 2.1 Populant

A Populant is a physical NPC agent that exists in the 3D scene tree and simultaneously participates in the management simulation. The management module's view of a Populant is a lightweight data component attached to the physical character scene.

#### 2.1.1 ManagementPopulantComponent

A component `Node` attached as a child to the physical NPC character scene root. It acts as the registration anchor between the 3D agent and the `ManagementAPI`.

**Script:** `res://scripts/characters/components/ManagementPopulantComponent.gd`
**Class:** `ManagementPopulantComponent`

**Variables:**

| Name | Type | Description |
| :--- | :--- | :--- |
| `character_id` | `int` | Unique identifier matching the NPC actor. Exported, set by designer or spawner. |
| `serving_lord_id` | `String` | The Lord faction this Populant is bound to (`"none"` if unrecruited). Exported. |
| `assigned_node_id` | `int` | The `ZoneNode` node_id this Populant resides at (`-1` if unassigned). Exported. |
| `qualification_profile` | `String` | The skill label registered into the workforce matching pool. Exported. |
| `current_3d_workplace_target` | `Marker3D` | Runtime reference to the physical 3D marker the NPC travels to when executing a job. Not exported. |

**Functions:**

* `_ready() -> void` (Private) — Calls `ServiceLocator.get_management_service()` and registers itself by calling `register_populant_component(self)` on the returned API.
* `_exit_tree() -> void` (Private) — Calls `ServiceLocator.get_management_service()` and unregisters itself by calling `unregister_populant_component(character_id)`.
* `is_assigned() -> bool` (Public) — Returns `true` if `serving_lord_id != "none"` and `assigned_node_id != -1`.

### 2.2 ZoneNode (Aggregate)

`ZoneNode` is the core domain aggregate of the management module. It encapsulates all data for a single territory: fertilities, stockpiles, buildings, resident workers, and job slots. It also owns the tick-time simulation subroutines for that territory.

The `ZoneNode` resource maps 1:1 to a `ZoneAnchor3D` spatial node placed in the 3D world (see Section 3 — Scene Hierarchy).

#### 2.2.1 ZoneNode (Resource)

**Script:** `res://scripts/resources/management/ZoneNode.gd`
**Class:** `ZoneNode`
**Extends:** `Resource`

**Signals:**

* `building_completed(node_id: int, building_id: String)` — Emitted when a building's `construction_progress` reaches `total_work_required`.
* `populants_starving(node_id: int, starving_count: int)` — Emitted when food runs out and some workers go unfed.

**Enums:**

* `SettlementTier { WILDERNESS, CAMP, FARM, MANOR, VILLAGE, TOWN }` — The administrative state of the node. A node at `WILDERNESS` skips all simulation ticks.

**Variables:**

| Name | Type | Description |
| :--- | :--- | :--- |
| `node_id` | `int` | Unique identifier for this node. Exported. |
| `node_name` | `String` | Display name. Exported. |
| `current_tier` | `SettlementTier` | Current administrative tier. Exported, default `WILDERNESS`. |
| `canopy_density` | `float` | Current forest volume (`0.0`-`1.0`). Exported, default `1.0`. |
| `fertilities` | `Dictionary` | Base resource fertility values (`"Timber"`, `"Cereals"`, `"Berries"`, `"Mushrooms"`). Exported. |
| `stockpile` | `Dictionary` | Current commodity amounts held by this node. Exported. |
| `allocated_construction_resources` | `Dictionary` | Materials earmarked for active construction orders but not yet consumed. Exported. |
| `max_storage_capacity` | `int` | Maximum total item count before auditing trims the excess. Exported, default `200`. |
| `buildings` | `Array[BuildingInstance]` | All construction orders — completed and in-progress — at this node. Exported. |
| `local_workers` | `Array[WorkerData]` | Structural links to every Populant assigned to reside here. Exported. |
| `manual_job_postings` | `Dictionary` | Player-opened environmental jobs (e.g., `{"forager": 2, "woodcutter": 1}`). Exported. |
| `_assigned_workers_by_job` | `Dictionary` | Runtime-only cache: job type to list of assigned character IDs. Populated and discarded each tick. Not exported. |

**Functions:**

* `has_available_resources(requirements: Dictionary) -> bool` (Public) — Returns `true` if `stockpile` contains at least as many of each resource as specified in `requirements`.
* `process_management_tick() -> void` (Public) — The master tick entry point. Immediately returns if `current_tier == WILDERNESS`. Otherwise executes the five simulation subroutines in order (see Section 4.1).
* `get_total_job_slots() -> Dictionary` (Public) — Calculates all open job slots: merges `manual_job_postings` with slots from buildings. Each incomplete building contributes `3` builder slots; each completed building contributes its `blueprint.max_jobs` under `blueprint.job_type`.
* `_resolve_workforce_allocation() -> void` (Private) — Clears the runtime cache and `assigned_worker_ids` arrays, then fills them by iterating `JobPriorities.HIERARCHY`, matching available workers to open slots greedily (see Section 4.2).
* `_deduct_sustenance_inventories() -> void` (Private) — Deducts food from `stockpile` for every resident in `local_workers`, cycling through `DietPriorities.CONSUMPTION_ORDER`. Emits `populants_starving` if any workers remain unfed.
* `_process_construction_progress() -> void` (Private) — Spends builder labor points (10 per builder per tick) on the first in-progress building. Marks complete when `construction_progress >= blueprint.total_work_required`, flushes `allocated_construction_resources`, emits `building_completed`.
* `_execute_production_and_labor() -> void` (Private) — Applies woodcutter and forager extraction formulas. Timber yield degrades `canopy_density`. Mushroom yield scales with remaining canopy.
* `_enforce_storage_caps() -> void` (Private) — Audits total stockpile count against `max_storage_capacity`. If over capacity, discards surplus items and emits a `push_warning`.
* `_get_assigned_count(job_type: String) -> int` (Private) — Returns the number of workers currently assigned to a given job type from `_assigned_workers_by_job`.

### 2.6 JobPriorities (System Constant)

A global constant class defining the job evaluation order for workforce allocation.

**Script:** `res://scripts/management/JobPriorities.gd`
**Class:** `JobPriorities`

**Constants:**

* `HIERARCHY: Array[String]` — Ordered job priority list. Workforce matching fills high-priority roles first:
  1. `"sustenance"` — Gathering food; prevents starvation.
  2. `"builder"` — Advances queued construction.
  3. `"woodcutter"` — Raw timber extraction.
  4. `"forager"` — Berry and mushroom collection.

**Functions:**

* `get_priority_weight(job_type: String) -> int` (Static Public) — Returns the index of `job_type` in `HIERARCHY`, or `999` if not found. Useful for external sorting.

## 3. Scene / Node Hierarchy

The management module has no standalone top-level scene. Its data and API live as resources and a script node; its spatial representation is anchored in the world scene.

### 3.1 ManagementAPI Node

`ManagementAPI.gd` is attached to a `Node` child of the `GameCoordinator`. It does **not** implement the `GameModule` contract and is not part of the daily simulation tick pipeline driven by the coordinator. It is the externally-facing command-and-query surface for UI, GOAP agents, and other systems.

> **Note on GameCoordinator Integration:** The management simulation tick is currently triggered by `ManagementAPI._on_day_changed()`, a handler designed to be connected to the `EventBus.day_changed` signal. That signal connection is pending the EventBus refactor (see `# TODO(event-system)` in `ManagementAPI.gd`).

```
GameCoordinator (Node)
└── ManagementAPI (Node)          <-- [Attached: ManagementAPI.gd]
                                       Registers to ServiceLocator on _ready()
```

### 3.2 ZoneAnchor3D (World Scene)

Each `ZoneNode` data resource maps 1:1 to a `ZoneAnchor3D` scene instance placed in the world. Planned — not yet implemented (see the implementation plan, Phase 3).

```
WorldSimulation (Node3D)
└── ZoneAnchor3D (Area3D)             <-- [Attached: ZoneAnchor3D.gd]
    @export var zone_data: ZoneNode   <-- the paired data resource
    ├── SelectionMesh (MeshInstance3D)
    └── CollisionShape3D
```

* **Visibility Filtering:** `toggle_management_view(is_active: bool)` enables collision layers and visual overlays when the management UI is open; disables them otherwise to eliminate selection overhead.
* **Building Spawning:** When `zone_data` emits `building_completed`, the anchor instantiates the matching visual 3D scene at a relative offset.

### 3.3 ManagementPopulantComponent (Character Scene)

Attached as a direct child of the physical NPC character root. Not a standalone scene.

```
NpcPeasant (CharacterBody3D)
├── StateMachine (Node)
├── HealthComponent (Node)
└── ManagementPopulantComponent (Node)   <-- [Attached: ManagementPopulantComponent.gd]
```

## 4. Systems

### 4.1 Master Simulation Tick (Cross-Script Process)

Every game day, `ManagementAPI._on_day_changed()` iterates every registered `ZoneNode` and calls `process_management_tick()`. Five subroutines inside execute sequentially:

```
Day Tick Signal
    └── ManagementAPI._on_day_changed()
            └── for each ZoneNode in _world_nodes:
                    └── ZoneNode.process_management_tick()
                            ├── Step 1: _resolve_workforce_allocation()
                            ├── Step 2: _deduct_sustenance_inventories()
                            ├── Step 3: _process_construction_progress()
                            ├── Step 4: _execute_production_and_labor()
                            └── Step 5: _enforce_storage_caps()
```

### 4.2 Workforce Allocation Algorithm

Runs as Step 1 of the simulation tick. Assigns every available `local_worker` to a job slot, prioritized by `JobPriorities.HIERARCHY`.

```
_resolve_workforce_allocation():
  1. Clear _assigned_workers_by_job cache.
  2. Clear all BuildingInstance.assigned_worker_ids arrays.
  3. Build full slot table via get_total_job_slots().
  4. For each job_type in JobPriorities.HIERARCHY:
      a. While open_slots > 0 and unassigned_workers is not empty:
          - Pop a worker from the front of the unassigned pool.
          - Append to _assigned_workers_by_job[job_type].
          - If job_type is a producer (not "builder"):
              Find the first matching completed building with capacity
              and append the worker ID to building.assigned_worker_ids.
```

### 4.3 Signal & Data Flow Boundaries

The management module emits signals for external observers but never directly calls other modules or the UI. See `architecture-overview.md` Section 3 for the global Call Down / Signal Up rule.

| Event | Signal | Intended Listener |
| :--- | :--- | :--- |
| Settlement tier changed | `ManagementAPI.settlement_tier_changed(node_id, tier_string)` | UI overlay, world renderer |
| Construction failed | `ManagementAPI.construction_failed_no_resources(node_id, building_id)` | UI feedback |
| Invalid command | `ManagementAPI.operation_failed(reason)` | UI error display |
| Building complete | `ZoneNode.building_completed(node_id, building_id)` | `ZoneAnchor3D` (spawns visual scene) |
| Workers starving | `ZoneNode.populants_starving(node_id, count)` | Social Module, Warfare/Health Module, GOAP |

### 4.4 NPC Integration (GOAP / State Machine)

When a Populant is assigned via `ManagementAPI.assign_populant_to_node()`, the API calls the private `_notify_assignment(character_id)`. This walks up from the `ManagementPopulantComponent` to the NPC actor and calls `assign_to_work()` if the method exists, triggering a state transition (idle/wandering to a work state). The NPC's GOAP brain or state machine then queries `ManagementAPI.get_character_assignment()` each tick to resolve its spatial movement target.

## 5. APIs

### 5.1 ManagementAPI (Externally Exposed Surface)

`ManagementAPI.gd` exposes the management module to the rest of the game. Other modules and the UI layer access `ZoneNode` data only through this boundary — never directly.

**Registration (Lifecycle):**

* `register_populant_component(comp: ManagementPopulantComponent) -> void` — Called by `ManagementPopulantComponent._ready()`. Adds the component to the internal `_registered_populants` dictionary.
* `unregister_populant_component(character_id: int) -> void` — Called by `ManagementPopulantComponent._exit_tree()`. Removes the component from the dictionary.

**Query API (Read-Only):**

* `get_character_assignment(char_id: int) -> Dictionary` — Returns `{"node_id": int, "job_type": String, "work_target_position": Vector3}` for the given character, or `{}` if unassigned. Position resolves from `current_3d_workplace_target` or falls back to the actor's world position.
* `get_node_canopy_density(node_id: int) -> float` — Returns the `canopy_density` of the specified node, or `0.0` if not found. Used by environmental rendering pipelines to update Terrain3D foliage.
* `get_node_inspection_data(node_id: int) -> Dictionary` — Returns a plain dictionary snapshot of a node's key fields for UI parsing. Returns `{}` if node not found.

**Command API (Mutation):**

* `establish_camp(node_id: int, owner_id: String = "player") -> void` — Transitions a `WILDERNESS` node to `CAMP`. Emits `operation_failed` if the node does not exist or is already settled. Emits `settlement_tier_changed` on success.
* `order_building(node_id: int, blueprint: BuildingData) -> bool` — Checks stockpile against `blueprint.build_cost`. On failure, emits `construction_failed_no_resources` and returns `false`. On success, deducts costs, moves them to `allocated_construction_resources`, creates a new `BuildingInstance`, and returns `true`.
* `recruit_populant_to_faction(character_id: int, lord_id: String = "player") -> void` — Sets `serving_lord_id` on the Populant's component. Prerequisite for node assignment.
* `assign_populant_to_node(character_id: int, target_node_id: int) -> bool` — Moves a faction-aligned Populant to a node. Removes them from any previous node's `local_workers`, updates `assigned_node_id`, appends a new `WorkerData` link, and calls `_notify_assignment()`. Returns `false` if the node does not exist or the Populant is unrecruited.

### 5.2 ServiceLocator Integration

`ManagementAPI` registers itself with the global `ServiceLocator` autoload during `_ready()`. Any module or component that needs management data calls:

```gdscript
var api = ServiceLocator.get_management_service()
```

The `ServiceLocator` asserts that the API is registered before returning it, catching out-of-order initialization at development time.
